# Implementation Audit (2026-03-03)

## 점검 범위
- 문서 기준: `README.md`, `claude.md`
- 코드 기준: `core/`, `ui/`, `news_scraper_pro.spec`
- 테스트 기준: `python -m pytest -q` (`97 passed, 5 subtests passed`)

## 핵심 결론
- 현재 기능은 전반적으로 안정적이며 회귀 테스트도 모두 통과합니다.
- 다만, 사용자 데이터 복원/상태 동기화/스레드 종료 경로에서 실제 운영 중 문제가 될 수 있는 잠재 리스크가 확인되었습니다.

## 주요 발견사항

### 1) [High] 설정-only 백업 복원 경로가 실패 상태로 고정될 수 있음
- 근거:
  - 백업 생성 UI는 `설정만` 백업을 허용함: `ui/dialogs.py:470-472`
  - 백업 목록에서도 DB 포함 여부를 표시함: `ui/dialogs.py:515`
  - 그러나 복원 예약 시 항상 `restore_db=True`로 고정 호출: `ui/dialogs.py:559`
  - pending 적용 로직은 `restore_db=True`인데 DB 백업 파일이 없으면 실패하고 pending을 유지함: `core/backup.py:271-278`
- 영향:
  - 사용자가 `설정만` 백업을 복원하려고 하면 시작 시 복원이 계속 실패할 수 있습니다.
  - `pending_restore.json`이 남아 재시작 때마다 동일 실패가 반복되는 UX/운영 이슈가 발생할 수 있습니다.
- 권장 조치:
  - 복원 예약 시 선택한 백업의 `include_db` 값을 읽어 `restore_db`에 그대로 전달.
  - 복원 확인 다이얼로그에 `설정만 복원 / 설정+DB 복원` 모드를 명시.

### 2) [Medium] 기사 열기/안읽음 액션에서 DB 실패 시 UI와 DB 상태가 불일치할 수 있음
- 근거:
  - `open` 액션에서 `update_status` 반환값을 확인하지 않고 UI 상태를 변경: `ui/news_tab.py:687-694`
  - `unread` 액션도 동일 패턴: `ui/news_tab.py:725-733`
  - 같은 파일의 공통 헬퍼(`_open_external_link_and_mark_read`)는 반환값 체크 후 상태 변경: `ui/news_tab.py:652-666`
- 영향:
  - DB 업데이트 실패(잠금/오류) 시 화면에는 읽음/안읽음이 바뀐 것처럼 보이지만 실제 저장 상태는 반영되지 않을 수 있습니다.
  - 다음 새로고침/재실행 시 사용자에게 상태가 되돌아간 것처럼 보일 수 있습니다.
- 권장 조치:
  - `open`, `unread` 경로도 DB 업데이트 성공 시에만 캐시/UI 갱신하도록 일원화.
  - 실패 시 토스트/상태바 경고를 추가해 사용자가 즉시 인지하도록 처리.

### 3) [Medium] 설정창 백그라운드 워커 종료가 비협조적 작업에서 불완전할 수 있음
- 근거:
  - `AsyncJobWorker`는 인터럽트 플래그를 작업 전/후에만 확인: `core/workers.py:45-52`
  - 설정창 API 검증 작업은 블로킹 `requests.get(..., timeout=10)` 호출: `ui/settings_dialog.py:630`
  - 설정창 종료 시 워커 정리는 최대 `400ms/800ms` 대기 후 종료 흐름 진행: `ui/settings_dialog.py:577-600`
- 영향:
  - 느린 네트워크/디스크 작업 중 설정창을 닫으면 워커가 완전히 종료되기 전에 다이얼로그 라이프사이클이 끝날 수 있습니다.
  - 간헐적으로 스레드 정리 경고나 종료 타이밍 이슈로 이어질 가능성이 있습니다.
- 권장 조치:
  - 설정창 전용 작업은 취소 토큰을 주기적으로 확인하는 협조적 작업 형태로 전환.
  - 종료 시 워커 상태를 명시적으로 확인하고, 미종료 시 콜백 무시뿐 아니라 안전한 후처리(분리 parent/수명 관리) 적용.

### 4) [Low] 탭 배지(미읽음 수)가 제외어 조건을 반영하지 않아 체감 수치가 어긋날 수 있음
- 근거:
  - 배지 계산은 `parse_tab_query`로 추출한 DB 키워드(첫 양키워드)만 사용: `ui/main_window.py:813-826`, `core/query_parser.py:20-24`
  - 집계 쿼리도 키워드 단위로만 계산: `core/database.py:809-833`
- 영향:
  - 예: `AI -광고` / `AI -코인` 탭은 실제 표시 결과가 달라도 배지 숫자가 동일하게 나올 수 있습니다.
- 권장 조치:
  - 배지 계산 시 탭별 제외어 조건까지 반영하는 집계 경로(또는 캐시 정책) 추가 검토.

## 추가해야 할 테스트(우선순위)
- [P1] 백업 복원 모드 회귀 테스트
  - `설정만` 백업 선택 시 `schedule_restore(..., restore_db=False)`가 호출되는지 검증.
- [P1] 뉴스 탭 상태 동기화 테스트
  - `update_status` 실패 시 `open/unread` 액션이 UI 캐시를 변경하지 않는지 검증.
- [P2] 설정창 워커 종료 안정성 테스트
  - 블로킹 작업 중 `closeEvent` 발생 시 잔존 워커/콜백 경쟁이 없는지 검증.
- [P2] 배지 정확도 테스트
  - 동일 DB 키워드 + 상이 제외어 탭에서 배지 정책(현재/개선안)을 명시적으로 고정.
